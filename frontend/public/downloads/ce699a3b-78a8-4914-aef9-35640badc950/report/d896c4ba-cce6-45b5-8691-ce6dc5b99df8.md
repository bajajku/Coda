# Building an Intelligent Accommodation Interview Engine: A Technical Deep Dive

### 1. Introduction: The Challenge of Automated Interviews
Conducting structured interviews for workplace or accessibility accommodations presents a unique architectural challenge. It requires a system that is rigid enough to ensure compliance with a defined taxonomy, yet flexible enough to handle the nuance of human variability. Traditional web forms often fail here; they are either too static, leading to user fatigue, or too unstructured, resulting in data that is difficult to process.

The "Accommodation Interview Backend" solves this by blending a high-performance FastAPI and WebSocket architecture with predictive AI. This engine allows for a dynamic, intelligent conversation that streamlines the transition from raw request to actionable accommodation. For developers and product owners, it offers a blueprint for building stateful, AI-assisted decision-making workflows that remain verifiable and robust.

### 2. The Core Foundation: Immutable Taxonomy and Persistent State
At the heart of the system is a design philosophy that separates business logic from session state to maximize performance. 

*   **Immutable Taxonomy:** We utilize an "Immutable Taxonomy" strategy. Rather than incurring the overhead of reloading decision trees or managing per-request deep copies through middleware, the system loads the taxonomy once into memory via `app/core/taxonomy.py`. This occurs during the FastAPI lifespan initialization in `main.py`, ensuring that every concurrent session references a high-speed, read-only model.
*   **Reliable State Management:** While the taxonomy is static, user sessions are dynamic and durable. We chose PostgreSQL to maintain the ground truth for every interaction. Managed through `app/db/session.py` and `db/taxonomy_db.py`, the system utilizes a **psycopg2 connection pool** to handle concurrent requests efficiently. This database layer ensures:
    *   **Request Persistence:** Initial text and uploaded files are stored immediately.
    *   **Answer Tracking:** Every step in the interview tree is persisted, allowing sessions to survive restarts or network interruptions.
    *   **Auditability:** The final calculated accommodations and the logic used to reach them are stored for future review.

### 3. The Interactive Workflow: From REST to WebSockets
The API surface, defined in `app/api/routes.py`, is split into two phases to optimize for bi-directional, stateful traversal.

**Step 1: Initiation (REST)**
The process begins with the `POST /v1/request` endpoint. This captures the user's primary request text and supporting documentation, returning a unique `request_id`. This initial "handshake" allows the system to prepare the state before moving to a real-time connection.

**Step 2: The Interview Loop (WebSockets)**
For the interview itself, we leverage `WS /v1/ws/session/{request_id}`. A Senior Architect understands that while long-polling or Server-Sent Events (SSE) can handle updates, WebSockets are the superior choice here for bi-directional stateful communication. This allows the backend to drive the interview flow based on the user's previous answers in a single, persistent connection.

The communication follows a strict JSON-based protocol:
*   `type: "llm_prefill_done"`: Emitted if the AI has successfully predicted answers based on initial data.
*   `type: "question"`: The backend sends a payload containing the `node_id`, question text, and valid options (e.g., "yes", "no").
*   `type: "answer"`: The client responds with the user's selection for a specific `node_id`.
*   `type: "accommodations"`: Once the traversal logic in `app/services/interview.py` is satisfied, the final results are delivered.

### 4. Intelligence Layer: LLM Prefill and Semantic Ranking
The system does not just react to user input; it anticipates it through a sophisticated intelligence layer.

*   **Gemini-Powered Prefill:** Using `app/services/llm_fill.py`, the system leverages the Gemini LLM to analyze the initial request and any uploaded documents—such as a `dataset/Physiotherapist Report.pdf`. The LLM attempts to predict answers for taxonomy nodes before the first question is even asked. This "human-first" prefill only executes on new sessions with no previous answers, ensuring that AI assistance never overwrites manual input.
*   **MiniLM Semantic Ranking:** Once the interview concludes, `app/services/semantic_similarity.py` uses MiniLM (a BERT-based transformer) to rank suggested accommodations. By calculating semantic similarity between the user's specific needs and the taxonomy outcomes, the engine ensures the most relevant suggestions appear first. Crucially, these BERT-based calls are scoped so that they only block the specific WebSocket connection they are serving, maintaining overall system responsiveness.

### 5. Developer Experience: Seeding and Testing
We prioritize a reproducible environment where the path from raw data to a running API is seamless. Orchestration is managed via a `Makefile`, which handles the Docker-based Postgres instance and triggers the `db/seed_taxonomy.py` script. This script synchronizes the database using `db/schema.sql` and the source `taxonomy.csv`.

For testing, the **Streamlit UI** (`streamlit_app.py`) provides a sandbox to test complex inputs, like physiotherapist reports, and observe the logic in real-time.

| Feature | Auto Mode | Manual Mode |
| :--- | :--- | :--- |
| **Workflow** | Runs LLM prefill, then automatically answers all **remaining** questions with "yes" or "no". | Runs LLM prefill, then displays individual questions with clickable buttons for user input. |
| **Developer Value** | Allows for rapid verification of the entire traversal path and final payload. | Allows for granular debugging and verification of specific taxonomy branches. |

### 6. Key Architectural Takeaways
Reflecting on the design of this engine, there are three critical lessons for full-stack architects:

1.  **Performance: The Lifespan Advantage.** By using the FastAPI lifespan startup for the "Immutable Taxonomy," we avoid the performance tax of per-request processing. Loading core business logic into memory at boot is a "pro-tip" for building high-throughput services.
2.  **Reliability: Connection Pooling and Persistence.** Moving beyond in-memory state is vital for production readiness. Using a persistent DB-backed state via a `psycopg2` pool ensures the system is concurrent, durable, and ready for scale.
3.  **Hybrid Logic Integration:** The most effective "Intelligent Engines" do not rely on AI alone. They use AI (Gemini) to provide "soft" assistance (pre-filling) while maintaining the integrity of a "hard-coded" taxonomy. This hybrid approach ensures the user experience is fluid while the results remain grounded in verifiable rules.

### 7. Conclusion: The Future of Accessible Interviews
The Accommodation Interview Backend demonstrates how modern web technologies and AI can be synthesized to solve complex administrative problems. By combining the speed of FastAPI, the real-time capabilities of WebSockets, and the predictive power of LLMs, we have created a framework that is both highly efficient and deeply accessible. This architecture provides a robust foundation for any organization looking to transform complex decision-making workflows into intuitive, AI-assisted experiences.
